Accessing and Utilizing Token-Level Probabilities in sherpa-onnx for Advanced Information-Theoretic AnalysisIntroductionThe evaluation of Automatic Speech Recognition (ASR) systems has traditionally been dominated by metrics centered on accuracy, most notably the Word Error Rate (WER).1 While indispensable for benchmarking transcription fidelity, WER provides a limited, post-hoc view of a model's performance. It quantifies the final outcome of the recognition process but offers little visibility into the internal decision-making dynamics, such as the model's confidence, its uncertainty when faced with ambiguity, or the distribution of probabilities across competing hypotheses. For researchers and engineers seeking to perform deeper diagnostics, analyze model behavior under challenging acoustic conditions, or develop more sophisticated confidence estimation measures, access to the underlying probability distributions generated by the model is paramount.A powerful lens for such analysis is provided by information theory. Tsallis entropy, a generalization of the standard Boltzmann-Gibbs (or Shannon) entropy, offers a flexible and potent metric for quantifying the uncertainty inherent in a probability distribution.2 Defined by the formula Sq​=kq−11−∑i​piq​​, where {pi​} is the set of probabilities for all possible outcomes and q is a tunable entropic index, Tsallis entropy can be adapted to emphasize different aspects of the probability distribution.5 Its calculation, however, is fundamentally dependent on having access to the full probability vector {pi​} over the model's vocabulary for each decision point.This requirement presents a significant challenge within the context of the sherpa-onnx framework. As a component of the next-generation Kaldi ecosystem, sherpa-onnx is engineered for high-performance, production-grade ASR inference.7 It is designed to be self-contained, operate entirely offline, and deployable across a vast range of platforms, from powerful x86-64 servers to resource-constrained embedded systems, mobile devices, and even WebAssembly environments.9 To achieve this remarkable efficiency and portability, its public-facing Application Programming Interfaces (APIs) are intentionally high-level. They are optimized to deliver the final transcription text swiftly, abstracting away the complex internal states and intermediate calculations, including the very token-level probability distributions necessary for entropy analysis. This creates a dilemma where the framework's primary strength—optimized performance—becomes an obstacle to deeper model introspection.This report provides a definitive and technically rigorous solution to this challenge. It presents a comprehensive methodology for accessing the requisite token-level log-probabilities from within the sherpa-onnx framework. The approach detailed herein involves a deep architectural analysis to identify where these probabilities are generated and stored, a practical, step-by-step guide to modifying the framework's C++ source code and C-API to expose this data, and a complete workflow for compiling a custom version of the library. Finally, it demonstrates how to interface with this modified library from Python to perform the necessary data transformations and calculate the Tsallis entropy of an ASR response, thereby enabling advanced, quantitative analysis of model uncertainty.The sherpa-onnx Inference and Decoding ArchitectureTo successfully extract the necessary probability distributions from sherpa-onnx, it is essential to first understand its internal architecture. The framework's design is a sophisticated interplay between a general-purpose neural network inference engine and a specialized set of speech recognition decoding algorithms. This separation of concerns, while designed for efficiency, also creates a clear and predictable interface where the desired probability data must exist. Understanding this interface is the key to locating and intercepting the data.The onnxruntime Backend and the k2-fsa CoreThe sherpa-onnx framework is built upon two foundational technologies: onnxruntime for neural network execution and the principles of the k2-fsa library for decoding.7sherpa-onnx explicitly uses onnxruntime as a replacement for PyTorch during inference, allowing it to execute pre-trained models that have been exported to the Open Neural Network Exchange (ONNX) format.9onnxruntime is a high-performance, cross-platform inference engine developed by Microsoft. Its core function is to take an ONNX model graph and execute it efficiently on a target device. It achieves this by partitioning the graph into subgraphs that can be run on the most suitable "execution providers," such as a CPU, a CUDA-enabled GPU, or other specialized accelerators, thereby maximizing computational throughput.12While onnxruntime handles the raw execution of the neural network components, the logic of how to combine their outputs into a coherent text transcription is governed by the principles of the k2-fsa project. k2 is a library designed to integrate Finite State Automaton (FSA) and Finite State Transducer (FST) algorithms directly into autograd-based machine learning toolkits like PyTorch.13 In the context of ASR, FSAs and FSTs are powerful mathematical constructs for representing the search space of possible transcriptions, incorporating constraints from acoustic models, pronunciation lexicons, and language models.A critical feature of the k2 framework is how it represents the connection between the neural network and the decoding graph. The weights, or "scores," associated with the arcs of an FSA in k2 are defined as log-probabilities.14 This is a deliberate design choice that allows the scores to be sourced directly from the output of a neural network's final log-softmax layer.14 This direct mapping is the fundamental link between the numerical output of the ONNX model and the structured search performed by the decoder.This architectural separation reveals a crucial point: the log-probability distribution is not an optional debugging value but rather the essential interface or data contract between the onnxruntime backend and the k2-based decoding logic. The process unfolds as follows:The sherpa-onnx application feeds audio features into onnxruntime.onnxruntime executes the relevant ONNX model graph (e.g., an encoder, decoder, and joiner for a Transducer model) and produces a final output tensor.12The sherpa-onnx C++ decoding logic receives this tensor and interprets its values as a distribution of log-probabilities over the vocabulary for a given time step.These log-probabilities are then used as the arc scores in a dynamically constructed FSA, which the decoder searches to find the most likely transcription path.Therefore, the full probability distribution over the vocabulary must be computed and exist in memory at each step of the decoding process. It is the fundamental unit of information passed from the neural model to the decoder. The high-level sherpa-onnx API simply consumes this information immediately to advance the decoding state and then discards it to conserve memory, hiding it from the end-user. The objective of this report is not to generate new information but to intercept this existing, transient data before it is discarded. This confirms the feasibility of the user's request and frames the technical approach as one of exposure and instrumentation, not novel calculation.Probability Generation in Transducer and CTC ModelsThe sherpa-onnx framework supports a diverse range of modern ASR architectures, with a primary focus on end-to-end models like Transducer (also known as RNN-T) and Connectionist Temporal Classification (CTC).9 These architectures, which include popular variants like Conformer and Zipformer, differ in how they generate the final probability distributions used for decoding.16Transducer (RNN-T) Models: These are autoregressive models that are typically composed of three distinct neural network components, each exported as a separate ONNX file:An Encoder network, which processes the input audio features and produces a high-level acoustic representation.A Prediction Network (often referred to as the "decoder" in this context), which acts as a language model, processing the sequence of previously emitted tokens to predict the next likely token.A Joiner network, which takes the output of the encoder (acoustic information) and the prediction network (linguistic context) and fuses them. The output of the joiner is the final tensor of log-probabilities over the entire vocabulary (including a special "blank" symbol) for the current audio frame and text context.18For Transducer models, the output of the joiner.onnx model is the precise location of the probability distribution needed for entropy calculations. It represents the model's final, combined judgment, conditioned on both what it has heard and what it has already transcribed. Therefore, any attempt to extract probabilities from a Transducer model must target the output tensor produced by the onnxruntime session run of the joiner graph.CTC Models: In contrast, CTC models are non-autoregressive.16 They typically consist of a single, powerful encoder network (e.g., a Conformer-CTC model) that processes the entire audio sequence.20 For each frame of the encoded audio, this network outputs a single vector of log-probabilities over the vocabulary, which, like the Transducer, includes a special "blank" token.21 The CTC decoding algorithm then finds the most probable path through this matrix of probabilities that corresponds to a valid transcription, collapsing repeated characters and removing blanks. For CTC-based models, the source of the probability distribution is the final output tensor of the main model.onnx graph.This architectural distinction is critical. This involves providing the theoretical basis for the calculation, interfacing with the custom C++ code from Python, and implementing the full data processing pipeline.Theoretical Basis and a Python Implementation of Tsallis EntropyThe Tsallis entropy, denoted as Sq​, is a generalization of the standard Boltzmann-Gibbs-Shannon entropy. For a discrete system with N possible states, each with a probability pi​ (where ∑i=1N​pi​=1), the entropy is defined as:Sq​({pi​})=kq−11−∑i=1N​piq​​where k is a positive constant, often set to 1 for simplicity, and q is a real number known as the entropic index.4The parameter q controls the behavior of the entropy measure.When q>1, the entropy calculation gives disproportionately more weight to the more probable states.When q<1, it gives more weight to the less probable (rare) states.In the limit as q→1, using L'Hôpital's rule, the Tsallis entropy converges to the standard Shannon entropy: S1​=−k∑i=1N​pi​ln(pi​).4This flexibility makes Tsallis entropy a valuable tool for nuanced analysis of model uncertainty. A robust and efficient Python implementation can be created using NumPy.Pythonimport numpy as np

def calculate_tsallis_entropy(probabilities: np.ndarray, q: float, k: float = 1.0) -> float:
    """
    Calculates the Tsallis entropy for a given probability distribution.

    Args:
        probabilities (np.ndarray): A 1D NumPy array of probabilities.
                                    Must sum to 1.
        q (float): The entropic index.
        k (float, optional): A positive constant. Defaults to 1.0.

    Returns:
        float: The calculated Tsallis entropy.
    """
    if not np.isclose(np.sum(probabilities), 1.0):
        raise ValueError("Probabilities must sum to 1.")

    if q == 1.0:
        # Handle the Shannon entropy case to avoid division by zero
        # Add a small epsilon to prevent log(0) for zero probabilities
        non_zero_probs = probabilities[probabilities > 0]
        return -k * np.sum(non_zero_probs * np.log(non_zero_probs))
    else:
        return k * (1 - np.sum(np.power(probabilities, q))) / (q - 1)

This yields a time-series of entropy values that provides a dynamic view of the model's confidence as the utterance unfolds. A sudden spike in entropy could indicate a moment of high acoustic ambiguity (e.g., a noisy segment, a cough) or linguistic ambiguity (e.g., the start of a complex or unpredictable phrase). Tracking entropy over time is therefore a powerful tool for pinpointing specific moments of uncertainty within an audio stream.Non-Streaming Recognition: In offline mode, the entire audio file is processed at once to produce a single, final transcription. The entropy calculated in this context represents an aggregate measure of the model's uncertainty over the entire utterance. While this single value lacks the temporal resolution of the streaming approach, it is useful for comparing the overall difficulty or ambiguity of different audio clips. For instance, a dataset of audio files could be ranked by their average transcription entropy to identify the most challenging examples for the model.The recommendation is to select the recognition mode that aligns with the analytical goal. For fine-grained, time-resolved analysis of model behavior, the streaming approach is superior. For high-level, comparative analysis of entire utterances, the non-streaming approach is more direct. For most academic and pure research purposes, this is the path of least resistance and greatest flexibility.Interpreting Entropy in ASR: From Numbers to MeaningCalculating an entropy value is only the first step; the true value lies in interpreting what that number signifies about the ASR model's state. A single entropy value, calculated from the probability distribution at a given time step, provides a snapshot of the model's certainty.Low Entropy: This indicates high model confidence. The probability mass is highly concentrated on a single token or a very small number of tokens. The model has a clear "winner" and perceives little ambiguity in its decision. This typically occurs during the recognition of common words in clear, high-quality audio.High Entropy: This indicates low model confidence and high ambiguity. The probability mass is spread more evenly across multiple competing tokens. The model is "unsure" which token to select. This can be triggered by several factors:Acoustic Ambiguity: Degraded audio quality, background noise, speaker disfluencies, or the presence of homophones (e.g., "to," "too," "two") can cause the model to assign similar probabilities to multiple phonetically plausible tokens.Linguistic Ambiguity: At points of high branching factor in a sentence, the language model component may find several next words to be grammatically and semantically plausible, leading to a flatter probability distribution.Out-of-Vocabulary (OOV) and Rare Words: When an ASR model encounters a word it has not seen frequently during training, it may struggle to transcribe it correctly. RNN-T models, in particular, have a tendency to delete or ignore words they are uncertain about.17 This uncertainty manifests as high entropy, as the model might distribute probability across several known sub-word units in an attempt to piece together the unfamiliar word.These interpretations open up numerous practical applications. While this is advantageous for deployment, it poses a significant challenge for researchers and advanced practitioners who require access to internal model states, such as token-level probability distributions, for deeper analysis. This report has demonstrated that this limitation is not insurmountable.A comprehensive, end-to-end methodology has been presented to successfully extract the necessary log-probability vectors. The core of this solution lies in a multi-stage process: first, conducting a thorough architectural analysis to identify that the required probability data serves as a fundamental interface between the onnxruntime backend and the k2-fsa decoding logic, and pinpointing its location in the C++ source code. Second, extending the framework's C-API to create a robust, language-agnostic access point for this data. Third, compiling a custom version of the sherpa-onnx library to incorporate these modifications. Finally, utilizing Python's ctypes library to interface with the custom build and implement the full data processing pipeline, including the critical transformation from log-probabilities to normalized probabilities, to accurately calculate Tsallis entropy.The central principle reinforced by this investigation is that the open-source nature and layered architecture of sherpa-onnx provide a clear and robust pathway for expert users to unlock capabilities beyond its default offerings. The presented technique of modifying the C-API is not merely a temporary workaround but a sustainable method for extending the framework's core functionality. This same approach can be readily adapted to expose other internal model states, such as attention weights or intermediate layer activations, positioning this guide as a foundational template for advanced framework customization. By bridging the gap between production-oriented optimization and the need for analytical introspection, this methodology empowers users to move beyond surface-level accuracy metrics and gain a more profound, quantitative understanding of ASR model behavior.
